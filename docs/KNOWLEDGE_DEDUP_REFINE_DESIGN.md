# 知识去重与精炼系统设计方案

## 一、背景与问题

在进行大规模网页爬取和知识提取时，我们面临一个核心问题：**不同来源的网页往往包含大量相似或重复的知识内容**。例如，在爬取10个关于数据流图的网页后，会发现：

- 多个网页都在介绍"外部实体"的定义，表述相近但略有差异
- "数据流命名规范"在不同文章中反复出现，有的详细有的简略
- 同一个案例（如图书馆管理系统）被多篇文章引用，细节程度不同

如果直接将这些内容全部存入知识库，会导致：
1. **存储冗余**：相同知识占用多份空间
2. **检索混乱**：Agent 查询时返回多个相似结果，难以选择
3. **质量参差**：同一概念有多个版本，无法确定哪个最准确完整
4. **维护困难**：更新一个概念需要修改多处

因此，我们需要建立一套知识去重与精炼机制，将原始提取的碎片化知识整合为结构清晰、内容精准的正式知识库。

---

## 二、系统架构

整个知识处理流程分为三个阶段：

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   原始提取层     │ ──▶ │   去重精炼层     │ ──▶ │   正式知识库    │
│   (artifacts)   │     │   (refiner)     │     │   (se_kb/*)     │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

**原始提取层**位于 `se_kb/artifacts/` 目录，按日期和时间组织，保留所有爬取的原始数据，用于溯源和审计。

**去重精炼层**是本方案的核心，负责识别重复内容、提取增量知识、执行智能融合。

**正式知识库**位于 `se_kb/diagrams/`、`se_kb/theory/` 等目录，存储经过精炼的高质量知识，供 Agent 直接调用。

---

## 三、去重策略

去重采用三层过滤机制，从粗到细逐步筛选：

### 3.1 第一层：语义相似度检测

使用文本嵌入模型将知识内容转换为向量表示，通过计算余弦相似度判断两份知识是否语义相近。当相似度超过设定阈值（建议 0.85）时，认为两份内容讨论的是同一主题。

这一层的作用是快速识别出"大致相同"的内容，避免对明显不相关的知识进行细粒度比较，提高处理效率。

### 3.2 第二层：结构化字段匹配

针对不同类型的知识，我们定义了关键标识字段。例如：

- 对于概念定义类知识（`diagrams.dfd.concepts`），关键字段是元素的 ID 和名称
- 对于规则类知识（`diagrams.dfd.rules`），关键字段是规则的 ID 和名称
- 对于案例类知识（`diagrams.dfd.examples`），关键字段是场景名称和领域

通过比较这些关键字段的重合度，可以精确判断两份知识是否描述的是同一个具体对象。例如，两份都在描述"外部实体"这个概念，即使表述不同，也应该被识别为重复。

### 3.3 第三层：增量知识检测

当确认两份知识描述同一对象后，还需要判断新内容是否带来了增量价值。增量可能包括：

- 新增的示例或用例
- 更详细的属性说明
- 补充的约束条件
- 额外的参考来源

只有当新内容存在有价值的增量时，才会触发知识融合；如果完全是已有内容的子集，则直接跳过。

---

## 四、精炼策略

当检测到有增量的重复内容时，系统会执行知识融合，将多份相关知识合并为一份更完整的版本。

### 4.1 融合原则

知识融合遵循以下原则：

**定义优先原则**：对于同一概念的定义，选择更准确、更完整的表述。如果两个定义各有侧重，则进行整合，形成涵盖两者要点的综合定义。

**示例合并原则**：将所有不重复的示例收集在一起。不同来源可能提供不同领域的案例，这些都是有价值的补充。

**属性取并集原则**：对于属性、特征、约束等列表型字段，取所有来源的并集。当同一属性有不同描述时，保留更详细的版本。

**来源溯源原则**：在融合后的知识中记录所有贡献来源的 URL，便于后续验证和引用。

### 4.2 融合实现

知识融合通过大语言模型完成。系统将已有知识和新知识一并提交给模型，要求其按照上述原则生成融合后的版本。模型能够理解自然语言表述的细微差异，判断哪些是真正的补充、哪些是重复表达，从而产出高质量的融合结果。

融合后的知识保持与原始知识相同的 JSON 结构，确保与知识库的其他部分兼容。

---

## 五、处理流程

### 5.1 离线批量精炼

这是主要的精炼方式，适用于定期处理累积的原始数据。流程如下：

1. **收集阶段**：扫描 `artifacts` 目录，读取指定时间范围内的所有提取结果
2. **分组阶段**：按知识类型（如 `diagrams.dfd.concepts`）对所有内容进行分组
3. **去重阶段**：在每个分组内部，使用三层过滤机制识别重复内容
4. **融合阶段**：对有增量的重复内容执行 LLM 融合，纯重复内容直接丢弃
5. **输出阶段**：将精炼后的知识写入正式知识库的对应目录

### 5.2 实时去重（可选）

在爬取提取的实时流程中加入去重检查，在写入 `artifacts` 之前先判断是否与已有知识重复。如果是纯重复则跳过写入；如果有增量则立即触发融合。这种方式可以避免产生大量冗余的原始数据，但会增加实时处理的延迟。

## 六、存储结构

### 6.1 原始数据（artifacts）

```
se_kb/artifacts/
└── 2025/
    └── 12/
        └── 11/
            └── 14_30/                    # 精确到分钟的时间目录
                └── blog.csdn.net/
                    └── 软件工程-数据流图.../
                        ├── metadata.json
                        ├── parsed.json
                        ├── document.md
                        ├── trace.json
                        └── dfd_concepts_csdn_xxx.json
```

原始数据按爬取时间组织，保留完整的提取上下文，用于溯源和问题排查。

### 6.2 正式知识库

```
se_kb/
├── diagrams/dfd/
│   ├── concepts/
│   │   └── dfd_concepts.json           # 融合后的概念定义
│   ├── rules/
│   │   └── dfd_rules.json              # 融合后的规则集
│   ├── examples/
│   │   ├── library_system.json         # 按场景组织的案例
│   │   ├── ecommerce.json
│   │   └── smart_community.json
│   ├── templates/
│   ├── validation/
│   └── levels/
└── theory/
    └── structured_analysis.json        # 融合后的理论知识
```

正式知识库按知识类型和主题组织，每个文件都是经过去重精炼的高质量内容。

## 八、质量保障

### 8.1 人工审核入口

系统在 Web 界面提供"待审核"视图，展示融合前后的对比，允许人工确认或调整融合结果。重要的知识更新需经人工审核后才正式生效。

### 8.2 版本控制

正式知识库的每次更新都记录版本信息，包括更新时间、涉及的源文件、融合的内容摘要。支持回滚到历史版本。

### 8.3 冲突处理

当融合过程中出现无法自动解决的冲突（如两个来源对同一属性给出矛盾的描述），系统会标记该冲突并保留两个版本，等待人工裁决。

---

## 九、实施计划

本方案建议分四个阶段实施：

**第一阶段**：实现离线批量精炼脚本，支持扫描指定日期的 artifacts 并按类型分组输出。

**第二阶段**：实现结构化字段匹配的去重逻辑，能够识别同类型知识中的重复条目。

**第三阶段**：接入 LLM 实现智能融合，将重复内容合并为精炼版本。

**第四阶段**：在 Web 界面添加"精炼"功能入口，支持一键执行去重精炼流程。

---

## 十、预期效果

实施本方案后，预期达到以下效果：

1. **存储效率提升**：正式知识库的体积减少 50% 以上，消除冗余内容
2. **知识质量提升**：每个概念、规则、案例都是融合多来源后的最完整版本
3. **检索精准度提升**：Agent 查询时返回唯一权威结果，无需在多个相似条目中选择
4. **可维护性提升**：更新知识只需修改一处，溯源信息完整可查

通过这套去重与精炼机制，我们能够将海量的原始爬取数据转化为结构化、高质量、可直接使用的知识资产，为 Agent 生成高质量 DFD 图提供坚实的知识基础。
